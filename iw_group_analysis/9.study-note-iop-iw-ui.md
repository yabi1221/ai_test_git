# iop-iw-ui 학습 노트

이 문서는 `iop-iw-ui`를 별도로 깊게 보기 위한 학습 노트다.  
목표는 "이 UI가 단순 화면이 아니라 포털 셸이라는 점"을 분명히 이해하는 것이다.

## 1. 이 프로젝트의 정체

`iop-iw-ui`는 IW 그룹의 사용자 진입점이다.  
사용자는 보통 이 프로젝트가 제공하는 화면부터 접속한다.

이 프로젝트는 단순 정적 프론트가 아니라 아래 역할을 함께 가진다.

- 화면 렌더링
- 로그인 시작점
- API 호출 클라이언트
- 업로드 시작점
- 외부 원격 프론트 모듈 연결점

즉, 포털 셸에 가깝다.

## 2. 사용 기술

### 언어
- TypeScript
- JavaScript

### 프레임워크/라이브러리
- React

- TailwindCSS

## 3. 왜 중요한가

이 프로젝트를 보면 아래가 바로 보인다.

- 사용자가 어떤 기능부터 쓰는지
- 로그인은 어디와 연결되는지
- API는 어디로 호출되는지
- 업로드는 어디로 보내는지
- 외부 원격 앱이 어떻게 묶이는지

즉, 전체 시스템의 출발점이다.

## 5. 이 프로젝트를 읽는 핵심 질문

- 사용자가 최초로 접속하는 화면은 어디인가?
- 로그인은 어디로 연결되는가?
- API 주소는 어디서 주입되는가?
- 업로드 주소는 어디서 주입되는가?
- 토큰은 어디서 저장/사용되는가?
- 외부 remote 앱은 어디서 불러오는가?

## 6. Keycloak 연동에서 보는 포인트

이 UI는 인증을 직접 처리하기보다, Keycloak 기반 로그인 흐름을 이용한다.

### 봐야 할 것
- Keycloak base URL
- realm
- client id
- redirect / callback 흐름
- 로그인 후 토큰 저장 방식
- refresh 동작 여부

### 이해해야 할 것
- 로그인은 UI 혼자 끝내지 않는다
- Keycloak과 연동돼야 전체 화면이 정상 동작한다
- 로그인 장애는 UI만 봐서는 안 된다

## 7. API 연동에서 보는 포인트

이 UI는 거의 모든 업무 데이터를 API에 의존한다.

### 봐야 할 것
- Axios 공통 클라이언트
- 요청 헤더 구성
- 공통 에러 처리
- 화면별 서비스 함수

### 이해해야 할 것
- UI는 데이터 소유자가 아니다
- 대부분의 업무 상태는 API 응답에 의존한다
- 따라서 UI 문제처럼 보여도 실제 원인은 API일 수 있다

## 8. 업로드 흐름에서 보는 포인트

이 UI는 파일 업로드를 API에 직접 보내지 않고 TUS 서버로 보낼 가능성이 크다.

흐름:
1. 사용자가 파일 선택
2. UI가 업로드 요청 시작
3. 업로드 대상은 `iop-iw-tus`
4. 업로드 완료 후 서버 쪽 후처리는 API가 담당

즉, 화면에서 첨부 기능을 본다면 API와 TUS를 같이 생각해야 한다.

## 9. 외부 원격 앱 연동 포인트

UI 환경변수에 아래 같은 값이 있다.

- `VITE_WIDGET_REMOTE_URL`
- `VITE_AI_ADMIN_REMOTE_URL`
- `VITE_CATALOG_REMOTE_URL`

이 의미는 아래와 같다.

- 포털 UI가 모든 화면을 혼자 들고 있지 않을 수 있다
- 외부 프론트 모듈을 런타임에 조합할 수 있다
- 단순 SPA보다 운영 구조가 복잡해질 수 있다

## 10. 이 프로젝트가 어려운 이유

- 인증 흐름이 프론트와 서버 양쪽에 걸쳐 있다
- API 호출이 많다
- 업로드 흐름이 별도 서버와 연결된다
- 원격 프론트 모듈 연동이 있다
- 환경변수 의존성이 높다

즉, 화면만 보면 되는 프로젝트가 아니다.
