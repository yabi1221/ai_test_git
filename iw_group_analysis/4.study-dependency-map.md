# IW 그룹 의존 관계 지도

이 문서는 IW 그룹 프로젝트들이 서로 어떻게 연결되는지, 어떤 방향으로 호출되고 의존하는지 한 번에 보기 위한 정리본이다.

## 1. 전체 구조 한눈에 보기

가장 단순한 구조는 아래와 같다.

`사용자 브라우저 -> iop-iw-ui -> Keycloak -> iop-iw-api -> DB / 외부 시스템`

파일 업로드 흐름은 아래처럼 별도 축이 추가된다.

`사용자 브라우저 -> iop-iw-ui -> iop-iw-tus -> iop-iw-api`

배포와 라우팅은 아래 저장소가 제어한다.

`iop-iw-manifest -> UI / API / TUS / GPKI / SCH 배포 정의`

특수 인증 축은 아래처럼 볼 수 있다.

`iop-iw-ui -> Keycloak(+ custom provider) -> 필요 시 iop-iw-gpki`

## 2. 프로젝트별 의존 방향

## `iop-iw-ui`

직접 의존하거나 호출하는 대상:
- `iop-iw-api`
- `keycloak`이 올라간 Keycloak 서버
- `iop-iw-tus`
- 외부 원격 프론트 모듈

설명:
- 사용자는 먼저 UI에 접속한다.
- UI는 로그인 시 Keycloak로 이동하거나 Keycloak 기반 인증 흐름을 탄다.
- 로그인 후 UI는 대부분의 업무 데이터를 API에서 가져온다.
- 대용량 파일 업로드는 TUS 서버로 보낸다.

## `iop-iw-api`

직접 의존하거나 호출하는 대상:
- MariaDB
- PostgreSQL
- Keycloak
- Kafka
- S3
- 외부 RAG API
- API Gateway / Catalog / Cache 계열

받아들이는 호출:
- `iop-iw-ui`의 업무 요청
- `iop-iw-tus`의 업로드 완료 훅

설명:
- 그룹의 중심 백엔드다.
- UI와 가장 강하게 결합돼 있다.
- 업로드 서버가 끝내지 못하는 업무 메타데이터 처리도 API가 담당한다.

## `iop-iw-tus`

직접 의존하거나 호출하는 대상:
- `iop-iw-api`

받아들이는 호출:
- `iop-iw-ui`의 업로드 요청

설명:
- TUS 서버는 파일을 받는 역할만 집중한다.
- 업로드 완료 후 내부 훅으로 API `/internal/tusd`를 호출한다.
- 따라서 저장은 TUS, 업무 후처리는 API로 분리된다.

## `iop-iw-gpki`

간접적으로 연결되는 대상:
- `keycloak`
- GPKI 라이브러리

설명:
- 일반 업무 흐름보다 특수 로그인 흐름에서 의미가 있다.
- Keycloak 확장의 `GpkiAuthenticator`와 연결될 가능성이 높다.
- 별도 WAR 웹앱으로 배포되므로 독립 서비스처럼 취급될 수 있다.

## `iop-iw-manifest`

관리 대상:
- `iop-iw-ui`
- `iop-iw-api`
- `iop-iw-tus`
- `iop-iw-gpki`
- `iop-iw-sch`

설명:
- 코드 호출 관계를 만들지는 않지만, 실제 운영 연결을 결정한다.
- 어떤 경로가 어느 서비스로 가는지 가장 정확하게 보여준다.
- 운영 환경에서 프로젝트들 사이의 실제 연결점은 manifest를 봐야 확정된다.

## `keycloak`

직접 의존하거나 연결되는 대상:
- Keycloak 본체
- 외부 사용자 저장소 DB
- `iop-iw-ui`
- `iop-iw-api`
- 필요 시 `iop-iw-gpki`

설명:
- 이 저장소는 인증 본체가 아니라 확장 코드 저장소다.
- 사용자 저장소 연동, GPKI 인증, MetaMask 인증, 세션 제한 정책을 확장한다.
- UI와 API 사이 로그인 흐름의 중심에 있다.

## `user-storage-jpa-master`

직접 의존하거나 연결되는 대상:
- Keycloak 본체
- MariaDB 사용자 저장소 DB

설명:
- 역할상 `keycloak`과 강하게 겹친다.
- 현재 운영 핵심보다는 비교/참고 또는 실험 축으로 보는 편이 맞다.

## 3. 호출 방향 요약

핵심 호출 방향만 뽑으면 아래와 같다.

- `브라우저 -> iop-iw-ui`
- `iop-iw-ui -> Keycloak`
- `iop-iw-ui -> iop-iw-api`
- `iop-iw-ui -> iop-iw-tus`
- `iop-iw-tus -> iop-iw-api`
- `iop-iw-api -> MariaDB`
- `iop-iw-api -> PostgreSQL`
- `iop-iw-api -> Keycloak`
- `iop-iw-api -> Kafka / S3 / RAG API`
- `keycloak -> 외부 사용자 저장소 DB`

## 4. 설정 기준 연결점

프로젝트 간 관계는 코드만 보면 헷갈릴 수 있다. 아래 설정값이 연결점을 가장 잘 보여준다.

## UI에서 보이는 연결점

- `VITE_API_BASE_URL`
- `VITE_UPLOAD_BASE_URL`
- `VITE_KEYCLOAK_BASE_URL`
- `VITE_KEYCLOAK_REALM`
- `VITE_KEYCLOAK_CLIENT_ID`
- `VITE_WIDGET_REMOTE_URL`
- `VITE_AI_ADMIN_REMOTE_URL`
- `VITE_CATALOG_REMOTE_URL`

의미:
- UI가 어떤 외부 서비스와 직접 대화하는지 드러난다.

## API에서 보이는 연결점

- `KEYCLOAK_*`
- `JWT_SECRET`
- `INTERNAL_API_KEY`
- `EXTERNAL_API_KEY`
- `TUS_SERVER_URL`
- `RAG_API_URL`
- `S3_*`
- `KAFKA_*`

의미:
- API가 어떤 인증 시스템, 업로드 서버, 외부 연동 시스템과 묶이는지 드러난다.

## Manifest에서 보이는 연결점

- `/iw` -> UI
- `/iw/api` -> API
- `/iw/public` -> API
- `/iw/files` -> TUS

의미:
- 운영 라우팅의 최종 연결을 보여준다.

## 5. 기능 기준 의존 관계

## 로그인 기능

흐름:
- 사용자 접속
- UI에서 로그인 시작
- Keycloak 인증
- 필요 시 custom provider / authenticator 동작
- UI가 토큰 획득
- API 호출 시 토큰 전달

관여 프로젝트:
- `iop-iw-ui`
- `keycloak`
- `iop-iw-api`
- 경우에 따라 `iop-iw-gpki`

## 문서/업무 기능

흐름:
- UI 화면에서 조회/수정 요청
- API Controller/Service/Mapper 처리
- DB 반영

관여 프로젝트:
- `iop-iw-ui`
- `iop-iw-api`

## 파일 업로드 기능

흐름:
- UI가 TUS 서버로 업로드 시작
- TUS가 파일 저장
- 업로드 완료 후 API 훅 호출
- API가 메타데이터/업무 후처리 수행

관여 프로젝트:
- `iop-iw-ui`
- `iop-iw-tus`
- `iop-iw-api`

## 배포 기능

흐름:
- Manifest에서 chart / values 관리
- 환경별 서비스 배포
- 라우팅과 서비스 연결 확정

관여 프로젝트:
- `iop-iw-manifest`
- UI / API / TUS / GPKI / SCH

## 6. 강결합 관계

특히 강하게 결합된 관계는 아래다.

- `iop-iw-ui <-> iop-iw-api`
- `iop-iw-ui <-> Keycloak`
- `iop-iw-tus -> iop-iw-api`
- `keycloak <-> 외부 사용자 저장소 DB`
- `iop-iw-manifest -> 전체 운영 연결`

설명:
- UI와 API는 사실상 한 서비스처럼 움직인다.
- 로그인은 UI와 Keycloak이 강하게 묶여 있다.
- 업로드는 TUS 혼자 끝나지 않고 API 후처리에 의존한다.
- Manifest 없이는 운영 연결을 정확히 확정하기 어렵다.

## 7. 느슨하거나 보조적인 관계

- `iop-iw-gpki`는 일반 업무 흐름보다 특수 인증 흐름에서 중요하다.
- `user-storage-jpa-master`는 현재 운영 핵심보다 참고/비교용일 가능성이 높다.
- `iop-iw-sch`는 manifest에만 보이며 현재 워크스페이스에는 없다.

## 8. 장애 추적 기준

어느 지점이 문제인지 찾을 때는 아래 순서로 보면 된다.

## 로그인 문제
- UI 환경변수 확인
- Keycloak client / realm 확인
- custom provider 배포 여부 확인
- API auth 처리 확인

## API 응답 문제
- UI 요청 URL 확인
- API 보안 설정 확인
- Controller / Service / Mapper 추적
- DB 연결 상태 확인

## 업로드 문제
- UI 업로드 URL 확인
- TUS 서버 동작 확인
- Hook 호출 확인
- API `/internal/tusd` 처리 확인

## 운영 라우팅 문제
- Manifest values 확인
- ingress / service 연결 확인
- 환경별 차이 확인

## 9. 한 줄 구조도

가장 실무적인 한 줄 구조도는 아래다.

`UI가 사용자 요청과 토큰을 들고 중심에서 움직이고, API가 업무 처리를 맡고, Keycloak이 인증을 맡고, TUS가 업로드를 맡고, Manifest가 운영 연결을 확정한다.`
