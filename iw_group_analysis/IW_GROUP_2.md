# iw_group 통합 심화 분석

## 목적

이 문서는 기존 그룹 공통 파생 문서들을 한 번에 묶은 통합 심화본이다.

## 1. 전체 구조

이 그룹은 크게 아래 4축으로 이해하면 된다.

1. `iop-iw-ui`: 사용자 화면
2. `iop-iw-api`: 핵심 업무 API
3. `iop-iw-tus`: 대용량 파일 업로드
4. `keycloak` + `manifest`: 인증과 배포 운영 축

핵심 흐름:

`브라우저 -> UI -> Keycloak -> API -> TUS/API 후처리`

## 2. 빌드와 기동 기준

정적 분석 기준으로는 Java 계열은 `Maven` 기준으로 보는 것이 가장 안전하다.

예외:

- `iop-iw-api`는 `Gradle` 흔적도 강하게 남아 있음
- 실제 운영/배포 기준은 `manifest`와 산출물 확인이 필요함

## 3. 개발/운영 범위

사용자 확인 기준으로 `dev` 실개발 축은 `msit`, `mois` 2개다.

판단:

- `mfds`는 잔존 흔적은 있으나 주개발 축으로 보기 어려움
- `prd`는 기관별 값 파일 복제 구조가 더 넓음

## 4. 가장 중요한 연동 축

### 인증

`UI -> Keycloak -> API auth -> 내부 사용자 판별`

### 업로드

`UI -> TUS -> /internal/tusd -> API 후처리`

### 문서/에디터

`UI 에디터 -> API 문서/과제/공유/댓글 -> 필요 시 RAG`

## 5. 주요 리스크

### manifest 정합성

- `api/sch -> tus-chart` 패턴 반복
- 기관별 값 파일 복붙 흔적
- `prd`에서 다른 기관 값이 섞인 흔적

### 인증 복잡도

- `Keycloak`
- API auth 계층
- 내부 사용자 판별
- 레거시 `ihb` 계열 공존

### 에디터/업로드 후처리

- UI 에디터
- TUS 훅
- 문서/첨부/댓글/권한 동시 변경

## 6. 실무 우선순위

처음 봐야 할 순서:

1. `iop-iw-api`
2. `iop-iw-ui`
3. `iop-iw-manifest`
4. `keycloak`
5. `iop-iw-tus`

## 7. 리팩토링 우선순위

1. `manifest` 정합성
2. API 인증/권한 계층 정리
3. UI 에디터/서비스 호출 방식 통일
4. 업로드 후처리 책임 분리
5. 레거시 `ihb` 축과 신규 축 경계 정리

## 결론

이 그룹은 `UI`, `API`, `업로드`, `인증`, `배포`가 강하게 결합된 포털 묶음이다.

실무적으로는 아래 두 줄만 먼저 기억하면 된다.

- 코드 복잡도 중심: `API + UI 에디터 + auth`
- 운영 리스크 중심: `manifest + Keycloak + 기관별 값`
