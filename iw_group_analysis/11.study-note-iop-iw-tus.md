# iop-iw-tus 학습 노트

이 문서는 `iop-iw-tus`를 별도로 깊게 보기 위한 학습 노트다.  
목표는 "이 프로젝트가 업무 서버가 아니라 업로드 전용 인프라 컴포넌트"라는 점을 분명히 이해하는 것이다.

## 1. 이 프로젝트의 정체

`iop-iw-tus`는 대용량 파일 업로드 전용 서버다.  
일반적인 Spring 애플리케이션처럼 업무 로직을 많이 담고 있는 프로젝트가 아니라, `tusd` 기반 업로드 서버를 배포하기 위한 저장소에 가깝다.

즉, 역할은 아래처럼 보면 된다.

- 파일 업로드 수신
- 대용량 업로드 처리
- 업로드 재개 지원
- 업로드 완료 이벤트 전달

## 2. 사용 기술

## 3. 왜 중요한가

이 프로젝트를 이해하면 아래가 풀린다.

- UI가 첨부파일을 왜 API가 아니라 다른 서버로 보내는지
- 대용량 업로드를 왜 별도 서버로 분리했는지
- 업로드 완료 후 왜 API가 다시 관여하는지
- 첨부 장애가 났을 때 어디부터 봐야 하는지

즉, 첨부/업로드 흐름을 이해하는 핵심 축이다.

## 5. 이 프로젝트를 이해하는 핵심 개념

## TUS Protocol

대용량 파일을 안정적으로 업로드하기 위한 프로토콜이다.  
중간에 끊겨도 이어서 올리는 resumable upload에 강하다.

이 그룹에서 중요한 이유:
- 일반 API 업로드보다 대용량 파일 처리에 유리하다
- 네트워크 불안정 상황에서 재시도가 쉽다

## tusd

TUS 서버 구현체다.  
이 프로젝트는 자체 애플리케이션 로직보다 `tusd` 실행 구성을 중심으로 움직인다.

## Hook

업로드 이벤트가 발생했을 때 다른 시스템에 알려주는 방식이다.  
이 프로젝트에서는 `pre-create`, `post-create`, `post-finish` 같은 훅이 중요하다.

## 6. 전체 흐름에서의 위치

이 프로젝트를 시스템 전체 흐름에서 보면 아래 위치에 있다.

`UI -> TUS 서버 -> API 후처리`

의미:
- 사용자는 UI에서 첨부를 시작한다
- 실제 파일 전송은 TUS 서버가 받는다
- 업로드 완료 후 업무 메타데이터 처리는 API가 맡는다

즉, 업로드 자체와 업무 후처리가 분리돼 있다.

## 7. API와의 관계

이 프로젝트는 혼자 완결되지 않는다.

핵심 관계:
- TUS는 파일을 저장한다
- API는 업로드 완료 후처리를 담당한다

주요 연결점:
- `TUSD_HOOKS_HTTP`
- API의 `/internal/tusd`

이 의미는 아래와 같다.

- 업로드 서버는 파일 저장 인프라 역할
- API는 첨부 메타데이터/업무 반영 역할

따라서 첨부 기능을 볼 때는 `iop-iw-tus`와 `iop-iw-api`를 같이 봐야 한다.

## 8. UI와의 관계

UI는 업로드 대상을 API가 아니라 TUS로 잡는다.

주요 연결점:
- `VITE_UPLOAD_BASE_URL`
- `VITE_TUS_API_KEY`

이 의미는 아래와 같다.

- 사용자가 화면에서 파일을 올리면
- 실제 업로드 요청은 TUS 서버로 간다
- 업로드 성공 후 최종 반영은 API 후처리를 거친다

## 9. 이 프로젝트가 어려운 이유

- 코드가 많지 않아서 오히려 역할이 추상적으로 느껴질 수 있다
- 일반 백엔드처럼 Controller / Service 구조가 없다
- 파일 업로드 자체와 업무 후처리가 분리돼 있다
- 장애 원인이 UI, TUS, API 중 어디인지 헷갈리기 쉽다

즉, 소스코드보다 "연결 구조"를 이해해야 한다.
