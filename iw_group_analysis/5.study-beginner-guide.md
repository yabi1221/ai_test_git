# IW 그룹 초급자 가이드

이 문서는 개발 경험이 많지 않거나, 이 프로젝트군을 처음 보는 사람이 전체 구조를 덜 막히게 이해하도록 돕기 위한 입문 안내서다.

## 1. 먼저 결론부터

이 그룹은 한 개의 프로그램이 아니라, 여러 프로젝트가 역할을 나눠서 함께 동작하는 구조다.

가장 단순하게 보면 아래처럼 이해하면 된다.

1. 사용자는 화면에 접속한다.
2. 로그인은 별도 인증 시스템이 처리한다.
3. 실제 업무 데이터 처리는 백엔드 서버가 담당한다.
4. 큰 파일 업로드는 전용 업로드 서버가 담당한다.
5. 실제 운영 연결은 배포 설정 저장소가 관리한다.

이걸 프로젝트 이름으로 바꾸면 아래다.

- 화면: `iop-iw-ui`
- 로그인: `keycloak`
- 업무 처리: `iop-iw-api`
- 파일 업로드: `iop-iw-tus`
- 운영 연결: `iop-iw-manifest`

## 2. 왜 프로젝트가 이렇게 나뉘어 있나

초급자가 가장 많이 막히는 지점은 "왜 하나로 안 만들고 나눴지?"다.

이유는 보통 아래와 같다.

### 화면과 서버는 역할이 다르다

- UI는 사용자에게 보이는 화면과 버튼을 담당한다.
- API는 실제 데이터 조회/저장 같은 업무 처리를 담당한다.

즉, "보여주는 것"과 "처리하는 것"이 다르기 때문에 나뉜다.

### 로그인은 공통 인증 시스템이 맡는 게 유리하다

- 여러 서비스가 각각 로그인 기능을 만들면 중복이 심하다.
- 그래서 Keycloak 같은 인증 서버를 두고 공통 로그인 체계를 만든다.

### 큰 파일 업로드는 별도 서버가 더 안정적이다

- 일반 API 서버가 대용량 업로드까지 다 맡으면 부담이 크다.
- 그래서 `iop-iw-tus` 같은 전용 업로드 서버를 분리한다.

### 운영 배포 정보는 코드와 분리하는 경우가 많다

- 소스코드 저장소와 운영 배포 설정 저장소를 나누면 환경별 관리가 쉬워진다.
- 그래서 `iop-iw-manifest` 같은 저장소가 따로 존재한다.

## 3. 프로젝트 이름을 보고 바로 역할 구분하는 법

## `iop-iw-ui`

이름에 `ui`가 들어가므로 화면 프로젝트다.

여기서 기대해야 하는 것:
- React 화면
- 로그인 버튼
- API 호출 코드
- 업로드 시작 코드

## `iop-iw-api`

이름에 `api`가 들어가므로 백엔드 서버다.

여기서 기대해야 하는 것:
- Controller
- Service
- DB 처리
- 인증 검증
- 문서/첨부/관리자 로직

## `iop-iw-tus`

이름만 보면 낯설지만, TUS는 업로드 프로토콜이다.

여기서 기대해야 하는 것:
- 파일 업로드
- 업로드 완료 이벤트
- 대용량 파일 처리

## `keycloak`

로그인과 사용자 인증 쪽 프로젝트다.  
단, 이 워크스페이스의 `keycloak`은 로그인 서버 본체가 아니라 "로그인 서버를 확장하는 코드" 저장소다.

## `iop-iw-manifest`

코드 저장소라기보다 배포 설정 저장소다.

여기서 기대해야 하는 것:
- Helm chart
- values 파일
- 라우팅 설정
- dev/prd 차이

## `iop-iw-gpki`

특수 인증 관련 프로젝트다.  
일반 업무 흐름보다 공공 인증서 로그인 같은 특수 흐름에서 중요하다.

## `user-storage-jpa-master`

이름만 보면 어려워 보이지만, 핵심은 Keycloak 사용자 저장소 확장 관련 참고 프로젝트라는 점이다.

## 4. 이 그룹을 볼 때 꼭 알아야 하는 최소 개념

## 프론트엔드와 백엔드

- 프론트엔드: 사용자가 보는 화면
- 백엔드: 화면 뒤에서 실제 처리하는 서버

이 그룹에서는:
- 프론트엔드 = `iop-iw-ui`
- 백엔드 = `iop-iw-api`

## 로그인 토큰

사용자가 로그인하면 서버가 "이 사용자는 인증됨"이라는 증표를 준다.  
그 증표가 토큰이다.

이 그룹에서는:
- 로그인은 Keycloak이 담당
- UI는 토큰을 받아 API 호출 시 같이 보냄

## 데이터베이스

사용자, 문서, 첨부 같은 실제 데이터를 저장하는 곳이다.

이 그룹에서는:
- MariaDB
- PostgreSQL

둘 다 등장한다.

## 업로드 서버

파일을 올리는 전용 서버다.  
특히 큰 파일은 일반 API보다 전용 업로드 서버가 더 적합하다.

이 그룹에서는:
- `iop-iw-tus`

## 배포 설정

코드를 어디에 어떻게 띄울지 정하는 설정이다.

이 그룹에서는:
- `iop-iw-manifest`

## 5. 초급자가 흔히 하는 오해

## 오해 1. `keycloak` 프로젝트가 로그인 서버 전체다

아니다.  
현재 분석 기준으로 이 저장소는 Keycloak 본체가 아니라 확장 코드 저장소다.

즉:
- Keycloak 서버 자체
- 그 위에 올리는 custom provider 코드

를 구분해서 봐야 한다.

## 오해 2. 파일 업로드도 그냥 API가 다 처리할 것이다

이 그룹은 그렇지 않다.

흐름은 대체로 아래에 가깝다.

- UI가 업로드 요청
- TUS 서버가 파일 저장
- 완료 후 API가 후처리

즉, 업로드와 업무 처리 책임이 분리돼 있다.

## 오해 3. manifest는 별로 중요하지 않다

실제로는 매우 중요하다.

코드만 보면 어느 서비스가 어디로 연결되는지 헷갈릴 수 있는데, 운영 기준 라우팅은 manifest가 가장 명확하게 보여준다.

## 오해 4. GPKI 프로젝트는 핵심 업무 서버일 것이다

아니다.

현재 정리 기준으로는 특수 로그인 보조 웹앱에 더 가깝다.

## 6. 가장 쉬운 읽기 순서

초급자는 아래 순서로 보면 된다.

### 1단계: UI 먼저 본다

이유:
- 사용자가 실제로 처음 만나는 지점이기 때문
- 어떤 API를 부르는지, 어떤 로그인 시스템을 쓰는지 드러나기 때문

확인할 것:
- API 주소
- Keycloak 주소
- 업로드 주소

### 2단계: API를 본다

이유:
- 실제 업무 처리가 여기 있기 때문
- 프로젝트 전체 복잡도의 중심이기 때문

확인할 것:
- Controller
- Service
- Mapper
- SecurityConfig

### 3단계: Keycloak을 본다

이유:
- 로그인 구조를 이해해야 전체 호출 흐름이 이어지기 때문

확인할 것:
- User Storage Provider
- Authenticator
- Event Listener

### 4단계: TUS를 본다

이유:
- 업로드 흐름은 일반 API 흐름과 다르기 때문

확인할 것:
- Dockerfile
- Hook 설정
- API 콜백 주소

### 5단계: Manifest를 본다

이유:
- 실제 운영에서 어느 경로가 어느 서비스로 가는지 확정해주기 때문

확인할 것:
- `/iw`
- `/iw/api`
- `/iw/files`

## 7. 초급자가 코드 읽을 때 보는 관점

코드를 한 줄씩 이해하려고 하면 금방 막힌다.  
대신 아래 질문으로 보면 된다.

### UI를 볼 때
- 이 화면은 누구와 통신하나?
- 로그인은 어디로 보내나?
- 데이터는 어느 API에서 가져오나?
- 파일은 어디로 올리나?

### API를 볼 때
- 이 요청은 어떤 Controller에서 받나?
- Service는 무슨 일을 하나?
- DB는 어디서 읽고 쓰나?
- 인증 검증은 어디서 하나?

### Keycloak을 볼 때
- 이 클래스는 로그인 단계에서 무슨 역할을 하나?
- 사용자 정보를 어디서 읽어오나?
- 세션 제한은 어디서 걸리나?

### Manifest를 볼 때
- 이 경로는 어느 서비스로 연결되나?
- dev와 prd의 차이는 무엇인가?

## 8. 이 프로젝트군에서 특히 어려운 부분

초급자가 특히 어렵게 느낄 지점은 아래다.

### 인증

이유:
- UI
- Keycloak
- API

세 군데가 같이 얽혀 있기 때문이다.

### 업로드 후처리

이유:
- 업로드는 TUS가 하고
- 업무 처리는 API가 하기 때문이다.

### 배포 구조

이유:
- 코드 저장소와 운영 저장소가 분리돼 있기 때문이다.

## 9. 처음에는 몰라도 되는 것

아래는 처음부터 깊게 파지 않아도 된다.

- MetaMask 인증 세부 구현
- QR 인증 세부 구현
- Kafka 연동 상세
- RAG API 상세
- Helm 템플릿 세부 문법

처음에는 전체 흐름을 잡는 게 더 중요하다.

## 10. 초급자용 한 줄 요약

이 그룹은 "화면, 로그인, 업무 처리, 파일 업로드, 배포"가 각자 다른 프로젝트로 나뉘어 협력하는 구조다.

## 11. 초급자가 마지막에 답할 수 있어야 하는 질문

- 사용자가 가장 먼저 접속하는 프로젝트는 무엇인가?
- 로그인은 누가 처리하는가?
- 실제 업무 데이터는 누가 처리하는가?
- 큰 파일 업로드는 누가 담당하는가?
- 운영 라우팅은 어디서 확인하는가?
- `keycloak` 저장소는 왜 Keycloak 본체와 다른가?

이 질문에 답할 수 있으면, 이 그룹의 기본 구조는 이미 잡힌 상태다.
