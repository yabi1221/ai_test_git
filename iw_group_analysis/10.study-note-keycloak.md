# keycloak 학습 노트

이 문서는 `keycloak` 프로젝트를 별도로 깊게 보기 위한 학습 노트다.  
목표는 "이 저장소가 Keycloak 본체가 아니라 인증 확장 코드"라는 점과, 실무에서 어디를 봐야 하는지 정리하는 것이다.

## 1. 이 프로젝트의 정체

`keycloak` 프로젝트는 Keycloak 서버 자체가 아니다.  
정확히는 `Keycloak 26`에 올리는 커스텀 Provider 확장 저장소다.

즉, 역할은 아래와 같다.

- 외부 사용자 저장소 연결
- 로그인 단계 확장
- 인증 방식 확장
- 세션 정책 확장

## 2. 사용 기술

### 언어
- Java 11

### 플랫폼
- Keycloak 26
- Keycloak SPI

### 구현 유형
- User Storage Provider
- Authenticator
- Event Listener Provider

### 확인된 확장
- JDBC 사용자 저장소 연동
- GPKI Authenticator
- MetaMask Authenticator
- Session Limit Event Listener

### 빌드
- Maven
- Shade Plugin

## 3. 왜 중요한가

이 프로젝트를 이해하면 아래가 풀린다.

- UI 로그인 흐름 뒤에서 어떤 인증 확장이 동작하는지
- 사용자 정보를 왜 Keycloak 내부가 아니라 외부 DB에서 가져올 수 있는지
- GPKI 같은 특수 인증이 어디서 붙는지
- 세션 제한 정책이 어디서 구현되는지

즉, 인증 구조의 핵심 확장점이다.

## 5. 이 프로젝트를 이해하는 핵심 개념

## User Storage Provider

Keycloak이 자체 사용자 저장소만 쓰지 않고, 외부 DB에서 사용자 정보를 읽을 수 있도록 확장하는 구조다.

이 그룹에서는 이 개념이 중요하다.  
왜냐하면 업무 시스템의 사용자 DB와 Keycloak 인증이 연결될 가능성이 크기 때문이다.

## Authenticator

로그인 흐름 중 특정 단계를 직접 구현하는 확장이다.

이 그룹에서는:
- GPKI 인증
- MetaMask 인증

같은 특수 방식이 여기에 해당한다.

## Event Listener

로그인, 로그아웃, 세션 생성 같은 이벤트를 감지해서 추가 정책을 적용하는 구조다.

이 그룹에서는 세션 수 제한이 대표적이다.

## 6. UI와 API 사이에서의 위치

이 프로젝트를 시스템 전체 흐름에서 보면 아래 위치에 있다.

`UI -> Keycloak(본체 + custom provider) -> API`

의미:
- UI는 로그인 요청을 시작한다
- Keycloak 본체는 기본 인증 플랫폼이다
- 이 저장소의 custom provider는 그 인증 과정을 확장한다
- API는 최종적으로 토큰을 검증하고 업무 접근을 통제한다

즉, 로그인 중심축 한가운데 있다.

## 7. GPKI와의 관계

이 프로젝트 안에 `GpkiAuthenticator`가 있다는 점은 중요하다.

의미:
- GPKI가 별도 외부 웹앱만의 문제가 아니다
- Keycloak 인증 흐름 안으로 들어와 있을 가능성이 높다
- 따라서 `iop-iw-gpki`와 같이 봐야 전체 구조가 보인다

## 8. user-storage-jpa-master와의 관계

`user-storage-jpa-master`도 역할이 많이 겹친다.

공통점:
- JDBC 사용자 저장소
- GPKI 계열 인증기
- 세션 제한

해석:
- `user-storage-jpa-master`는 실험/이전 버전 또는 확장판일 가능성이 있다
- `keycloak`은 현재 운영용 축약/정리 버전일 가능성이 있다

즉, 둘을 비교하면 인증 구조의 진화 방향을 추정할 수 있다.

## 9. 이 프로젝트가 어려운 이유

- Keycloak 자체를 모르면 이해가 어렵다
- 일반 Spring 프로젝트와 구조가 다르다
- SPI 개념이 낯설다
- 로그인 단계가 눈에 바로 보이지 않는다
- 외부 DB와 연결되는 지점이 숨어 있다

그래서 초급자는 "이 클래스가 어떤 순간에 실행되는가"를 중심으로 읽어야 한다.
