# iop-iw-api 학습 노트

이 문서는 `iop-iw-api`를 별도로 깊게 보기 위한 학습 노트다.  
목표는 "이 프로젝트가 왜 복잡한지"와 "어디부터 읽어야 하는지"를 분명히 잡는 것이다.

## 1. 이 프로젝트의 정체

`iop-iw-api`는 IW 그룹의 핵심 업무 백엔드다.  
UI가 호출하는 대부분의 업무 기능이 이 프로젝트에 들어 있다.

단순 CRUD 서버가 아니라 아래 기능이 한데 모여 있다.

- 인증/보안
- 문서 처리
- 첨부 처리
- 사용자/권한
- 관리자 기능
- 배치/스케줄링
- 외부 시스템 연동
- 업로드 후처리

즉, 큰 모놀리식 업무 서버로 보는 편이 맞다.

## 3. 왜 중요한가

이 프로젝트를 이해하면 아래가 같이 풀린다.

- UI가 실제로 무엇을 호출하는지
- 로그인 후 어떤 인증 검증이 있는지
- 문서/첨부/관리자 기능이 어디서 처리되는지
- 업로드 후처리가 어디서 일어나는지
- DB와 외부 시스템이 어떻게 연결되는지

즉, 그룹 전체의 중심 축이다.

## 5. 패키지 읽는 법

이 프로젝트는 패키지가 많기 때문에 순서를 정하지 않으면 금방 길을 잃는다.

처음에는 아래 순서로 본다.

1. `auth`
2. `document`
3. `attachment`
4. `admin`
5. `user`

이유:
- 인증과 실제 업무 흐름이 만나는 지점이 많기 때문
- 문서/첨부가 UI와 가장 강하게 연결되기 때문

## 6. Controller -> Service -> Mapper 흐름

이 프로젝트를 읽을 때는 항상 아래 흐름으로 따라간다.

1. Controller에서 URL을 받는다.
2. Service에서 업무 로직을 처리한다.
3. Mapper에서 SQL을 연결한다.
4. XML 매퍼에서 실제 쿼리를 본다.

중요한 점:
- 클래스 이름만 보고 이해하려 하지 말고, 한 요청을 끝까지 따라가야 한다.
- 처음부터 모든 Controller를 보지 말고 화면 하나와 연결된 것만 추적하는 편이 낫다.

## 7. 인증 구조에서 보는 포인트

이 프로젝트에서 인증은 별도 부가 기능이 아니라 핵심 축이다.

### 봐야 할 것
- JWT가 어디서 읽히는가
- Keycloak 토큰 검증이 어디서 되는가
- 외부 API Key 검증은 어떻게 처리되는가
- 인증 예외 처리는 어디서 되는가

### 이해해야 할 것
- 로그인 자체는 Keycloak 중심이지만
- 업무 API 보호는 결국 이 API 서버가 책임진다

즉:
- Keycloak은 로그인/토큰 발급
- API는 토큰 검증/업무 접근 보호

## 8. 데이터 구조에서 보는 포인트

### 메인 DB
- MariaDB

### 공통 DB
- PostgreSQL

### 왜 중요하나
- 하나의 요청이 어느 DB를 타는지 구분해야 한다
- 스케줄링/공통 사용자/조직 정보 쪽은 별도 축일 수 있다
- 장애가 나면 DB 어느 쪽 문제인지 먼저 구분해야 한다

## 9. 업로드 후처리 포인트

`iop-iw-api`는 업로드 서버는 아니지만, 업로드 흐름에서 매우 중요하다.

흐름:
1. UI가 TUS 서버로 업로드
2. TUS 서버가 파일 저장
3. 업로드 완료 후 API `/internal/tusd` 호출
4. API가 메타데이터/업무 후처리

즉, TUS만 보면 업로드 흐름이 끝나지 않는다.

## 10. 외부 연동 포인트

이 프로젝트는 단순 내부 DB만 쓰지 않는다.

확인된 외부 연동:
- Keycloak
- Kafka
- S3
- RAG API
- API Gateway
- Catalog / Cache
- 메일/메시지 계열

초기 학습에서는 세부 구현까지 파지 말고, "이 프로젝트가 외부 시스템 허브 역할도 한다"는 점만 잡으면 된다.

## 11. 이 프로젝트가 어려운 이유

- 패키지 수가 많다
- 엔드포인트 수가 많다
- 인증 로직이 얽혀 있다
- DB가 하나가 아니다
- 업로드 후처리까지 맡는다
- 외부 연동도 많다
- 빌드 체계가 Maven/Gradle 혼재다

즉, 기능이 너무 많이 한 프로젝트에 들어 있다.
