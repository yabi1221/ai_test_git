# IW 그룹 학습 질문집

이 문서는 IW 그룹 프로젝트를 공부한 뒤 이해도를 점검하거나, 스터디/인수인계 때 질문 목록으로 사용할 수 있도록 정리한 문서다.

## 1. 전체 구조 질문

- 이 그룹의 7개 프로젝트는 각각 어떤 역할을 하나?
- 이 그룹에서 가장 핵심적인 프로젝트 3개를 꼽는다면 무엇이며, 왜 그런가?
- `iop-iw-ui`, `iop-iw-api`, `keycloak`, `iop-iw-tus`, `iop-iw-manifest`를 한 문장으로 연결해서 설명할 수 있는가?
- 사용자가 처음 접속하는 지점은 어디인가?
- 실제 업무 데이터 처리는 어느 프로젝트가 담당하는가?
- 운영 라우팅은 어느 저장소에서 확인하는가?

## 2. UI 관련 질문

- `iop-iw-ui`는 단순 화면 프로젝트인가, 아니면 포털 셸 역할까지 하는가?
- UI가 직접 통신하는 대상은 무엇인가?
- `VITE_API_BASE_URL`은 무엇을 의미하는가?
- `VITE_UPLOAD_BASE_URL`은 무엇을 의미하는가?
- `VITE_KEYCLOAK_BASE_URL`, `REALM`, `CLIENT_ID`는 왜 필요한가?
- UI에서 토큰은 어떻게 API 호출에 전달될 가능성이 높은가?
- UI가 외부 원격 프론트 모듈과 연결된다는 말은 무슨 뜻인가?

## 3. API 관련 질문

- `iop-iw-api`는 어떤 기술 스택을 사용하는가?
- `iop-iw-api`가 대형 모놀리식 구조에 가깝다고 본 이유는 무엇인가?
- 이 프로젝트에서 MyBatis가 중요한 이유는 무엇인가?
- `pom.xml`과 `build.gradle`이 공존하는 것은 어떤 의미인가?
- API는 어떤 종류의 업무를 담당하는가?
- 인증/보안 관련 처리는 어느 계층에서 일어날 가능성이 높은가?
- API가 TUS 서버와 연결되는 지점은 어디인가?
- 왜 이 API는 단순 CRUD 서버보다 더 복잡한가?

## 4. 인증 관련 질문

- 이 그룹에서 로그인은 누가 담당하는가?
- `keycloak` 프로젝트는 Keycloak 본체인가, 아니면 다른 것인가?
- Keycloak SPI란 무엇인가?
- User Storage Provider는 무엇을 위한 기능인가?
- Authenticator는 로그인 흐름에서 어떤 역할을 하는가?
- Event Listener는 어떤 운영 정책 구현에 쓰일 수 있는가?
- UI, Keycloak, API는 인증 흐름에서 각각 어떤 역할을 가지는가?
- 왜 인증 구조가 초급자에게 특히 어렵게 느껴질 수 있는가?

## 5. GPKI 관련 질문

- GPKI는 이 그룹에서 어떤 종류의 인증 흐름으로 보이는가?
- `iop-iw-gpki`는 핵심 업무 서버인가, 특수 로그인 웹앱인가?
- `keycloak` 안의 `GpkiAuthenticator`와 `iop-iw-gpki`는 어떤 관계일 가능성이 높은가?
- GPKI 관련 기능을 볼 때 왜 한 프로젝트만 보면 안 되는가?

## 6. 업로드 관련 질문

- `iop-iw-tus`는 일반 API 서버와 무엇이 다른가?
- TUS 프로토콜은 왜 필요한가?
- UI는 파일 업로드를 어디로 보내는가?
- 파일 업로드 완료 후 왜 API가 다시 관여하는가?
- `/internal/tusd` 경로는 어떤 용도로 보이는가?
- 업로드 장애가 나면 어떤 순서로 추적해야 하는가?

## 7. 데이터베이스 관련 질문

- `iop-iw-api`는 왜 MariaDB와 PostgreSQL을 같이 사용할 가능성이 있는가?
- 다중 DataSource 구조는 왜 복잡도를 높이는가?
- Keycloak 사용자 저장소가 업무 DB와 연결될 가능성이 있다는 말은 무슨 뜻인가?
- `user-storage-jpa-master`의 docker-compose 정보가 왜 중요한 단서가 되는가?

## 8. Manifest 관련 질문

- `iop-iw-manifest`는 왜 중요한가?
- 코드 저장소만 보고는 알기 어려운 것을 manifest가 무엇으로 보완해주는가?
- `/iw`, `/iw/api`, `/iw/files` 라우팅은 각각 무엇을 뜻하는가?
- `dev`와 `prd` 차이는 보통 어디서 확인하는가?
- `iop-iw-sch`가 manifest에는 있는데 워크스페이스에는 없는 것은 어떤 의미일 수 있는가?
- `path: tus-chart` 같은 값이 API 쪽에 들어 있다면 왜 의심해볼 만한가?

## 9. 비교 질문

- `keycloak`과 `user-storage-jpa-master`는 어떤 점이 비슷한가?
- 두 저장소 중 현재 운영용에 더 가까워 보이는 쪽은 어디이며, 그 이유는 무엇인가?
- `iop-iw-ui`와 일반적인 단순 SPA의 차이는 무엇인가?
- `iop-iw-tus`와 일반적인 파일 업로드 API의 차이는 무엇인가?
- `iop-iw-gpki`와 `keycloak`의 역할 차이는 무엇인가?

## 10. 실무 추적 질문

- 로그인 버튼을 눌렀을 때 내부적으로 어떤 흐름이 이어질까?
- 문서 조회 화면을 클릭했을 때 어떤 프로젝트들이 관여할까?
- 첨부 업로드가 실패했을 때 어떤 저장소부터 볼 것인가?
- 사용자 권한이 잘못 적용됐을 때 어떤 프로젝트와 설정을 함께 봐야 할까?
- 운영에서 `/iw/api` 호출이 이상한 서비스로 가는 것 같다면 어디를 먼저 확인할까?

## 11. 설명형 질문

- 이 그룹을 처음 보는 사람에게 1분 안에 구조를 설명해보라.
- 초급자에게 왜 `UI -> API -> Keycloak -> TUS -> Manifest` 순서로 보라고 권할 수 있는가?
- 왜 이 그룹은 "기능 이름"보다 "흐름" 기준으로 이해해야 하는가?
- 왜 업로드는 TUS와 API를 같이 봐야 하는가?
- 왜 인증은 UI와 Keycloak과 API를 같이 봐야 하는가?

## 12. 심화 질문

- 이 구조에서 가장 강결합된 프로젝트 관계는 무엇인가?
- 어느 부분이 향후 장애 포인트가 되기 쉬운가?
- 어떤 설정 불일치가 운영 장애로 이어질 가능성이 높은가?
- 어떤 저장소는 현재 운영 핵심이고, 어떤 저장소는 참고/보조 성격으로 보이는가?
- 빌드 체계 혼재나 중복 저장소 존재가 유지보수에 어떤 부담을 줄 수 있는가?

## 13. 답할 수 있으면 기본 구조를 이해한 질문

아래 질문에 막힘 없이 답할 수 있으면 기본 이해는 잡힌 것이다.

- 사용자는 어디로 접속하는가?
- 로그인은 누가 처리하는가?
- 업무 API는 누가 처리하는가?
- 업로드는 누가 처리하는가?
- 업로드 완료 후 후처리는 누가 하는가?
- 실제 운영 라우팅은 어디서 확인하는가?
- GPKI는 어느 프로젝트들과 같이 봐야 하는가?
- Keycloak 사용자 저장소는 왜 별도 확장 코드가 필요한가?

## 14. 한 줄 마무리 질문

이 그룹을 설명하는 가장 중요한 질문 하나는 이것이다.

- "사용자 요청이 들어와서 실제 처리되고 다시 화면에 보이기까지, 어떤 프로젝트들이 어떤 순서로 관여하는가?"
