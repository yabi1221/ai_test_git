## 사용 방법

- 처음부터 전부 읽지 말고, 순서대로 체크하면서 본다.
- 각 항목은 "파일 확인", "구조 이해", "연결 관계 확인" 기준으로 구성했다.
- 막히면 항상 `UI -> API -> Keycloak -> TUS -> Manifest` 흐름으로 다시 돌아온다.

## 1. 전체 구조 체크

- [ ] 7개 프로젝트 이름을 외우지 않고 역할 기준으로 구분할 수 있다.
- [ ] `iop-iw-ui`가 사용자 진입점이라는 것을 이해했다.
- [ ] `iop-iw-api`가 핵심 업무 백엔드라는 것을 이해했다.
- [ ] `keycloak`이 인증 확장 저장소라는 것을 이해했다.
- [ ] `iop-iw-tus`가 업로드 전용 서버라는 것을 이해했다.
- [ ] `iop-iw-manifest`가 배포 정의 저장소라는 것을 이해했다.
- [ ] `iop-iw-gpki`와 `user-storage-jpa-master`는 보조/특수 성격의 저장소라는 것을 이해했다.

## 2. UI 체크리스트

### 먼저 볼 것
- [ ] `iop-iw-ui`의 환경변수 구조를 확인했다.
- [ ] `VITE_API_BASE_URL`의 의미를 이해했다.
- [ ] `VITE_UPLOAD_BASE_URL`의 의미를 이해했다.
- [ ] `VITE_KEYCLOAK_BASE_URL`, `REALM`, `CLIENT_ID`의 의미를 이해했다.

### 코드에서 볼 것
- [ ] API 호출 공통 설정 파일을 찾았다.
- [ ] 인증 관련 서비스 파일을 찾았다.
- [ ] 토큰 첨부 로직을 확인했다.
- [ ] 토큰 만료 시 refresh 처리 흐름을 확인했다.
- [ ] 업로드 요청 시작 지점을 찾았다.

### 이해해야 할 것
- [ ] UI가 직접 호출하는 대상이 API인지, Keycloak인지, TUS인지 구분할 수 있다.
- [ ] UI가 단순 화면이 아니라 포털 셸 역할이라는 점을 이해했다.
- [ ] 외부 원격 앱 URL이 왜 필요한지 설명할 수 있다.

## 3. API 체크리스트

### 먼저 볼 것
- [ ] `MainApplication.java`를 확인했다.
- [ ] 어떤 Spring 기능이 활성화돼 있는지 확인했다.
- [ ] `application.yml` 계열 설정 파일 구조를 확인했다.

### 보안/인증
- [ ] `SecurityConfig`를 확인했다.
- [ ] JWT 필터 또는 토큰 처리 클래스를 확인했다.
- [ ] Keycloak 검증 관련 클래스를 찾았다.
- [ ] API Key 인증 관련 로직이 있는지 확인했다.

### 구조
- [ ] Controller -> Service -> Mapper 흐름을 하나 이상 따라가 봤다.
- [ ] `document`, `attachment`, `auth`, `admin` 중 하나를 실제로 추적했다.
- [ ] MyBatis XML 매퍼와 Java 코드 연결 방식을 이해했다.
- [ ] 다중 데이터소스 설정을 확인했다.

### 외부 연동
- [ ] MariaDB 연결 지점을 확인했다.
- [ ] PostgreSQL 연결 지점을 확인했다.
- [ ] S3, Kafka, RAG API 같은 외부 연동 설정을 확인했다.
- [ ] `TUS_SERVER_URL` 또는 `/internal/tusd` 관련 로직을 확인했다.

### 이해해야 할 것
- [ ] API가 인증, 문서, 첨부, 관리자 기능을 모두 가진 대형 백엔드라는 점을 설명할 수 있다.
- [ ] API가 TUS 후처리도 맡고 있다는 점을 설명할 수 있다.

## 4. Keycloak 체크리스트

### 먼저 볼 것
- [ ] `keycloak` 프로젝트가 Keycloak 본체가 아니라 확장 Provider 저장소라는 점을 이해했다.
- [ ] `pom.xml`을 보고 Keycloak 버전과 Java 버전을 확인했다.

### 핵심 클래스
- [ ] `JdbcUserStorageProviderFactory`를 확인했다.
- [ ] `GpkiAuthenticatorFactory`를 확인했다.
- [ ] `MetaMaskAuthenticatorFactory`를 확인했다.
- [ ] `SessionLimitEventListenerFactory`를 확인했다.

### 이해해야 할 것
- [ ] User Storage Provider가 무엇인지 설명할 수 있다.
- [ ] Authenticator가 인증 흐름 어느 지점에서 동작하는지 설명할 수 있다.
- [ ] Event Listener가 어떤 운영 정책에 쓰이는지 설명할 수 있다.
- [ ] UI와 API 사이에서 Keycloak이 어떤 위치에 있는지 설명할 수 있다.

### 비교 체크
- [ ] `user-storage-jpa-master`와 기능 겹침을 확인했다.
- [ ] 두 저장소 중 어떤 쪽이 운영용에 더 가까운지 추정 근거를 정리할 수 있다.

## 5. TUS 체크리스트

### 먼저 볼 것
- [ ] `Dockerfile`을 확인했다.
- [ ] `DockerfileK8s`를 확인했다.
- [ ] 업로드 기본 경로가 어디인지 확인했다.
- [ ] 최대 파일 크기 제한을 확인했다.

### 핵심 흐름
- [ ] TUS 프로토콜이 resumable upload 용도라는 점을 이해했다.
- [ ] 이벤트 훅 종류를 확인했다.
- [ ] 업로드 완료 후 API 콜백 주소를 확인했다.

### 이해해야 할 것
- [ ] TUS는 파일 저장 인프라이고, 업무 메타데이터 처리는 API가 맡는다는 점을 설명할 수 있다.
- [ ] 장애가 나면 `UI -> TUS -> API 후처리` 순서로 추적해야 한다는 점을 이해했다.

## 6. Manifest 체크리스트

### 먼저 볼 것
- [ ] 어떤 chart들이 있는지 확인했다.
- [ ] `dev`, `prd` 구성을 확인했다.
- [ ] `iop-iw-api`, `iop-iw-ui`, `iop-iw-tus`, `iop-iw-gpki`, `iop-iw-sch`를 찾았다.

### 라우팅 확인
- [ ] `/iw`가 어디로 가는지 확인했다.
- [ ] `/iw/api`가 어디로 가는지 확인했다.
- [ ] `/iw/files`가 어디로 가는지 확인했다.
- [ ] `/iw/public`이 어디로 가는지 확인했다.

### 운영 관점 확인
- [ ] values 파일로 환경 차이를 비교했다.
- [ ] ArgoCD application YAML 경로를 확인했다.
- [ ] `path: tus-chart` 같은 이상 설정이 있는지 확인했다.

### 이해해야 할 것
- [ ] 실제 서비스 연결 관계는 manifest가 가장 정확히 보여준다는 점을 이해했다.
- [ ] 코드 저장소와 운영 배포 저장소를 분리해서 봐야 한다는 점을 이해했다.
